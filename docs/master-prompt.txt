[SECTION 4: SUPABASE SETUP RULES]
─────────────────────────────────────────────────────────────────────────────────

4-A. Environment Variables
     Always use these exact names:
       NEXT_PUBLIC_SUPABASE_URL=
       NEXT_PUBLIC_SUPABASE_ANON_KEY=
     NEVER use SUPABASE_SERVICE_ROLE_KEY in application code.
     Service role is only for one-off migration scripts, never in Next.js routes.

4-B. Client Creation
     Browser (Client Components):
       import { createBrowserClient } from "@supabase/ssr";
       createBrowserClient(process.env.NEXT_PUBLIC_SUPABASE_URL!, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!)

     Server (Server Components / Route Handlers):
       Use a shared createSupabaseServerClient() that handles cookies via next/headers.
       See lib/supabase/server.ts pattern below.

4-C. Middleware (REQUIRED — without this, auth cookies break on every navigation)
     Create src/middleware.ts that calls supabase.auth.getUser() to refresh the
     session on every request. Match all protected routes with the matcher config.

4-D. Auth Pattern
     - Email/password signup + email confirmation (Supabase defaults)
     - After login, redirect to /(protected)/dashboard
     - Server Components check getServerUser() and redirect("/login") if null
     - NEVER trust client-side auth state for security decisions

4-E. Database Schema Design
     Rules:
     - Every table has: id (uuid default gen_random_uuid()), created_at (timestamptz default now())
     - Use snake_case for all column names
     - Every user-owned table has: user_id uuid references auth.users(id)
     - Enable Row Level Security (RLS) on every table
     - Write RLS policies: user can only SELECT/INSERT/UPDATE/DELETE their own rows
     - Use foreign key constraints and ON DELETE CASCADE where appropriate

     Example minimal schema for any "entities + events" app:
       CREATE TABLE entities (
         id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
         user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE,
         name text NOT NULL,
         metadata jsonb DEFAULT '{}',
         created_at timestamptz DEFAULT now()
       );
       ALTER TABLE entities ENABLE ROW LEVEL SECURITY;
       CREATE POLICY "owner_all" ON entities FOR ALL USING (auth.uid() = user_id);

       CREATE TABLE events (
         id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
         entity_id uuid REFERENCES entities(id) ON DELETE CASCADE,
         event_type text NOT NULL,
         payload jsonb DEFAULT '{}',
         ai_analysis jsonb,              ← store AI output as jsonb
         created_at timestamptz DEFAULT now()
       );
       ALTER TABLE events ENABLE ROW LEVEL SECURITY;
       CREATE POLICY "owner_all" ON events FOR ALL
         USING (entity_id IN (SELECT id FROM entities WHERE user_id = auth.uid()));

4-F. Querying
     - Always use .select() with explicit column list (not .select("*")) in prod
     - Chain .order("created_at", { ascending: false }) as default
     - Use .single() when expecting exactly one row; handle null explicitly
     - For related data: prefer explicit JOINs via !inner over nested selects

5-B. Prompt Engineering Rules
  - Always include a ROLE line at the top: "You are a [role] that [purpose]."

  - Include fallback: if parsing fails, store raw text and surface error gracefully
  - Keep system prompts < 2000 tokens; user prompts < 4000 tokens for Flash

5-D. AI Analysis Storage Pattern
  Store all AI outputs as jsonb columns alongside the source data row.
  This enables: re-displaying without re-calling AI, audit trails, debugging.
  Example: calls table has summary text, ai_analysis jsonb (structured parsed output).

5-E. RAG (Retrieval-Augmented Generation) Pattern
  When the AI needs to answer questions about user's own data:
  1. Fetch relevant rows from Supabase (last N records, filtered by entity)
  2. Serialize to a compact string: `[${date}] ${eventType}: ${summary}`
  3. Inject into prompt as "CONTEXT" section
  4. Ask AI to cite which context items it used
  5. Return citations alongside the answer for credibility

[SECTION 6: NEXT.JS APP ROUTER PATTERNS]
─────────────────────────────────────────────────────────────────────────────────

6-A. Server Component First
  Write every page as an async Server Component.
  Fetch ALL data at the page level and pass as props to Client Components.
  Never use useEffect for data fetching; never use SWR/React Query unless offline
  sync is required.

6-B. Client Islands Pattern
  "Client Islands" = Client Components embedded inside Server Components.
  Rule: A Client Component receives all data it needs as props from its Server parent.
  Rule: A Client Component handles only UI interactions (clicks, forms, modals).
  Pattern:
    // page.tsx (Server Component)
    const data = await fetchData(supabase);
    return <MyFeatureClient data={data} />;

    // my-feature-client.tsx (Client Component)
    "use client";
    export function MyFeatureClient({ data }: { data: MyData[] }) { ... }

6-C. Route Handlers (API Routes)
  Always create route.ts (not route.js) in src/app/api/
  Always authenticate first: check user with supabase.auth.getUser()
  Always validate input with zod
  Return NextResponse.json() with appropriate status codes
  Never return 200 for errors

6-D. Loading States
  Use React Suspense + loading.tsx for page-level loading.
  For component-level: use Suspense with a skeleton fallback.
  For AI responses: show streaming text or skeleton while awaiting.

6-E. Error Handling
  Create error.tsx for page-level error boundaries.
  API routes: always return { error: string } in error cases.
  Client: show toast or inline error, never silent failures.

6-F. Optimizations (add to next.config.ts)
  experimental: {
    optimizePackageImports: ["@supabase/supabase-js", "@supabase/ssr", "@google/generative-ai"],
  }

8-B. SMS Pattern (Solapi / Korea)
  POST to Solapi API with HMAC-SHA256 auth.
  Always store: recipient, message, status, provider_message_id, sent_at in DB.
  Retry logic: max 3 attempts, exponential backoff 1s/2s/4s.

8-C. TTS Call Pattern
  Synthesize script → POST to Solapi voice call endpoint.
  Store: call_id, script, duration, keypad_response, call_status in DB.
  Webhook: Create /api/webhooks/call-result/route.ts to receive async completion.
  After call ends: trigger AI analysis of call summary via background job.

8-D. Mock Mode
  In mock mode: still create DB records (delivery, call) with status="mock".
  This allows the full app flow to be tested without real API costs.
  Mock responses should simulate realistic results (keypad: "1" or "2", random).

[SECTION 9: AI ANALYSIS PIPELINE]
─────────────────────────────────────────────────────────────────────────────────

This is the core of what makes SilverLink-quality apps special.
After any user event (call, form submission, log entry), run an AI analysis pipeline:

Step 1: Trigger
  After saving raw data to DB → POST to /api/ai/analyze (fire-and-forget or await)

Step 2: Context Building
  Fetch last 10 relevant records for this entity from DB.
  Build a context string: summarize each record in 1–2 lines.

Step 3: Analysis Prompt
  System: "You are an expert analyst in [domain]. Your job is to detect signals
           in [entity] behavior data and classify them by category and severity."
  User:   "CONTEXT:\n{context}\n\nNEW EVENT:\n{eventDescription}\n\n
           OUTPUT JSON with: categories[], severity (high/medium/low),
           summary, suggestions[]"

Step 4: Parse & Store
  Parse JSON from response.
  Store in ai_analysis jsonb column of the relevant table.
  If severity==="high": create a separate alerts record for immediate surfacing.

Step 5: Surface
  Alerts page: show all high-severity items, sorted by created_at desc.
  Dashboard: show count badge of unread alerts.
  Detail page: show per-entity analysis inline.

[SECTION 10: MULTI-ROLE / CASEWORKER PATTERN]
─────────────────────────────────────────────────────────────────────────────────

Any app that serves both individual users and supervisors/managers needs two modes.

Implementation:
  1. Store role in Supabase user_metadata: { role: "individual" | "manager" }
  2. After role change, call supabase.auth.refreshSession() to update JWT cookie
     (without this, Server Components still read the old role from the stale cookie)
  3. Manager dashboard: fetch all entities, sort by risk score descending
  4. Risk score calculation:
       - Low engagement count (missed check-ins)
       - Unread high-severity alerts count
       - Trend direction (score going down over last 4 weeks)
  5. Risk flags on entity cards:
       - CRITICAL: score < 40 AND missed last 3 events
       - WARNING: score 40–60 AND declining trend
       - INFO: unread alerts >= 3

[SECTION 11: SCORING & TREND METRICS]
─────────────────────────────────────────────────────────────────────────────────

Any engagement/health metric follows this pattern:

Weekly Score Calculation:
  call_response_rate = (calls_responded / calls_total) * 100   [weight: 70%]
  link_response_rate = (links_clicked / links_sent) * 100      [weight: 30%]
  weekly_score = min(100, (call_rate * 0.7) + (link_rate * 0.3))

Score Labels:
  >= 70: "Active" / "Good" (emerald)
  40-69: "Normal" / "Okay" (amber)
  < 40:  "Low" / "At Risk" (rose)

Trend Chart (8-week view):
  Store weekly_scores table: entity_id, week_start (date), score, calls_total, calls_responded
  Recompute weekly score after every new call response.
  Chart: Recharts or plain SVG polyline (no external chart library needed for simple cases).

  Group events by week.
  Color-code by event_type: 📞 blue (calls), 🔔 rose (alerts), 📋 purple (reports).
  Show a trend chart above the timeline (calls/alerts/score per week).

EXAMPLE 1: HR Onboarding Tracker
  "신입 직원의 30일 온보딩 체크리스트를 HR 담당자가 AI로 자동 모니터링하는 앱.

import { createServerClient } from "@supabase/ssr";
import { NextResponse, type NextRequest } from "next/server";

export async function middleware(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request });
  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return request.cookies.getAll(); },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) => request.cookies.set(name, value));
          supabaseResponse = NextResponse.next({ request });
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          );
        },
      },
    }
  );
  await supabase.auth.getUser();
  return supabaseResponse;
}

import { createServerClient } from "@supabase/ssr";
import { cookies } from "next/headers";

export async function createSupabaseServerClient() {
  const cookieStore = await cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll(); },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            );
          } catch { /* Server Component — ignore */ }
        },
      },
    }
  );
}
