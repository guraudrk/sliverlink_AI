## 세션 6: 앱 기능 전면 수정 — OAuth·데이터·UI 4개 이슈 해결

### 요청과 목표

- 빌드 완료 후 테스트에서 발견된 4가지 문제 동시 수정
  1. Google 로그인 → "invalid flow state, no valid flow state found" 오류
  2. 어르신 등록 탭에서 등록 후 홈 화면으로 돌아오면 목록이 갱신 안 됨
  3. 기존 테스트 계정으로 로그인해도 홈 화면에 어르신 목록이 안 보임
  4. 홈 화면 낮/밤 모드 전환 버튼과 기능별 바로가기 그리드 메뉴 누락

### 구현 및 변경 내용

**1. Google OAuth "invalid flow state" 수정**

기존 구조는 `login.tsx`에서 `WebBrowser.openAuthSessionAsync` 완료 후 `exchangeCodeForSession`을 직접 호출하는 방식이었다. 이 방식은 Android에서 브라우저 복귀 직후 AsyncStorage의 PKCE code verifier가 아직 준비되지 않은 시점에 교환을 시도해 state mismatch 오류를 유발했다.

해결책: `exchangeCodeForSession` 호출을 `login.tsx`에서 제거하고, 대신 `_layout.tsx`에 `Linking.addEventListener("url", ...)` 리스너를 등록해 딥링크 콜백을 받는 즉시 교환하도록 이동했다. 앱이 완전히 포그라운드로 복귀한 후 AsyncStorage 접근이 안정화된 시점에 실행되므로 state 불일치가 발생하지 않는다. cold start(앱이 닫힌 상태에서 딥링크로 열리는 경우)는 `Linking.getInitialURL()`로 별도 처리했다.

**2. 어르신 목록 미반영 수정 (`useFocusEffect`)**

`index.tsx`의 데이터 조회가 컴포넌트 마운트 시 한 번만 실행되는 `useEffect`에 묶여 있었다. 탭 간 이동 시 이미 마운트된 컴포넌트는 재실행되지 않아 등록 탭에서 돌아와도 목록이 갱신되지 않았다. `expo-router`의 `useFocusEffect`로 교체해 탭이 포커스를 얻을 때마다 Supabase에서 재조회하도록 변경했다.

**3. 다크/라이트 모드 테마 시스템 (`ThemeProvider`)**

`lib/theme-context.tsx`를 새로 만들어 LIGHT/DARK 두 가지 팔레트를 정의하고 `AsyncStorage`에 사용자 선택을 저장한다. `_layout.tsx` 최상위에 `ThemeProvider`로 앱 전체를 감싸고, 각 화면에서 `useTheme()` 훅으로 `colors`, `isDark`, `toggleTheme`에 접근한다.

홈 화면 헤더 우상단에 Sun/Moon 토글 버튼을 추가했고, 탭바(`_layout.tsx`)도 `useTheme()`를 통해 배경색·보더색·아이콘 색상이 테마에 따라 동적으로 변하도록 수정했다.

**4. 빠른 메뉴 그리드**

홈 화면 스탯 패널 아래에 8개 항목 4열 반응형 그리드를 추가했다: 통화 녹음, AI 분석, 어르신 등록, 일정 관리, 안전 알림, 통계, AI 비서, 설정. 각 항목은 아이콘 + 레이블 조합으로 해당 탭 또는 화면으로 라우팅된다.

### 문제와 해결 과정

- **"invalid flow state" 원인 분석**: OAuth PKCE 흐름에서 `expo-auth-session`이 생성한 code verifier를 AsyncStorage에 저장하는 시점과, 브라우저에서 앱으로 복귀하는 시점 사이에 타이밍 경합이 발생한다. Android Chrome Custom Tab이 닫히면서 앱이 포그라운드로 오는 직후 `exchangeCodeForSession`을 즉시 호출하면 verifier가 아직 쓰기 완료되지 않아 실패할 수 있다. Linking 이벤트 기반으로 처리를 뒤로 미루면 이 경합을 피할 수 있다.
- **`useEffect` → `useFocusEffect` 전환**: React Navigation / Expo Router에서 탭은 처음 방문 후 언마운트되지 않기 때문에 `useEffect`는 최초 마운트 한 번만 실행된다. 탭 포커스 복귀 시 재실행이 필요한 데이터 조회는 반드시 `useFocusEffect`를 사용해야 한다.

### 기술적 의사결정

- **Linking 기반 OAuth vs. WebBrowser 반환값 기반**: WebBrowser의 반환값에서 URL을 파싱해 `exchangeCodeForSession`을 호출하는 방식이 더 단순하지만 Android에서 타이밍 문제가 발생한다. Linking 이벤트 기반은 코드가 약간 분산되지만 신뢰성이 높다.
- **테마 Context vs. 외부 라이브러리**: `react-native-paper`, `tamagui` 등 UI 라이브러리 없이 최소한의 Context + AsyncStorage로 구현해 번들 크기 증가 없이 다크/라이트 전환을 제공했다.

### 테스트 및 검증

- 로컬 코드 변경만 완료된 상태이며, 실제 앱에서의 동작 검증은 versionCode 5 EAS 빌드 완료 후 가능하다.
- TypeScript 타입 오류 없이 저장되었고 컴파일 오류 없음(런타임 테스트 미완료).

### 변경 파일 (silverlink-mobile)

- `lib/theme-context.tsx`: LIGHT/DARK 팔레트, ThemeProvider, useTheme 훅 — 신규 생성
- `app/_layout.tsx`: Linking 딥링크 핸들러 추가, ThemeProvider 래핑
- `app/(auth)/login.tsx`: Google OAuth에서 `exchangeCodeForSession` 제거, browser 열기만 담당
- `app/(tabs)/index.tsx`: `useFocusEffect` 데이터 재조회, 빠른 메뉴 그리드, 다크/라이트 토글 버튼
- `app/(tabs)/_layout.tsx`: `useTheme()` 연동으로 탭바 동적 테마 적용

### 미해결 사항 및 다음 작업

## 세션 7: 네이티브 안드로이드 백그라운드 통화 감지 구현

### 요청과 목표

- 앱이 백그라운드에 있을 때도 등록된 어르신과의 통화를 자동으로 감지하고 싶다는 사용자 요청
- 통화 종료 후 통화 메타데이터(어르신 ID, 통화 시간)를 Supabase에 자동 저장
- 통화 후 삼성 AI 요약 공유를 유도하는 푸시 알림 표시
- 앱에서 "전화 연결" 버튼으로 발신할 때도 동일하게 기록

### 구현 및 변경 내용

**아키텍처: 두 가지 통화 경로 처리**

1. **앱에서 발신 (주 경로)**: 어르신 카드 팝업의 "전화 연결" 버튼 → JS에서 `setPendingOutgoingCall(elderId, elderName)` 호출 → SharedPreferences에 60초 유효 pending 저장 → 안드로이드가 OFFHOOK 감지 시 pending을 active로 승격 → IDLE(통화 종료) 시 DB 저장 + 알림
2. **어르신 수신 전화**: BroadcastReceiver가 RINGING 시 `EXTRA_INCOMING_NUMBER`를 읽어 등록된 어르신 번호와 매칭 → OFFHOOK 시 통화 시작 시간 기록 → IDLE 시 처리

**네이티브 코틀린 파일 (신규)**

- `CallStateReceiver.kt`: `TelephonyManager.ACTION_PHONE_STATE_CHANGED` BroadcastReceiver. RINGING → OFFHOOK → IDLE 상태 전이 추적. Supabase REST API(`/rest/v1/call_recordings`)에 `status: "detected"`로 직접 HTTP POST. `NotificationCompat`으로 통화 완료 알림 표시
- `CallDetectionModule.kt`: React Native 네이티브 모듈. `syncElders(eldersJson, token, url, key, userId)`, `setPendingOutgoingCall(elderId, name)` 두 메서드를 `@ReactMethod`로 노출. SharedPreferences에 저장
- `CallDetectionPackage.kt`: `ReactPackage` 구현체로 `CallDetectionModule` 등록

**기존 파일 수정**

- `MainApplication.kt`: `add(CallDetectionPackage())` 추가
- `AndroidManifest.xml`: `READ_PHONE_STATE`, `READ_CALL_LOG`, `POST_NOTIFICATIONS` 권한 추가; `<receiver android:name=".CallStateReceiver" android:exported="true">` 등록
- `app.json`: `android.permissions` 배열에 3개 권한 추가
- `src/modules/call-detection.ts` (신규): JS 브릿지. `requestCallDetectionPermissions()`, `syncElders()`, `setPendingOutgoingCall()` 함수 export
- `app/_layout.tsx`: `requestCallDetectionPermissions()` 세션 시작 시 호출
- `app/(tabs)/index.tsx`: `fetchData` 완료 후 `syncElders()` 호출; 전화 연결 버튼에 `setPendingOutgoingCall()` 추가
- `android/app/build.gradle`: versionCode 9 → 10, versionName 1.0.8 → 1.0.9

### 문제와 해결 과정

- **발신 시 번호 불명 문제**: Android 10+에서 `ACTION_PHONE_STATE_CHANGED` 브로드캐스트에 발신 번호가 포함되지 않는다. 앱에서 전화를 걸기 직전에 어르신 정보를 SharedPreferences에 저장하는 `setPendingOutgoingCall` 방식으로 우회했다.
- **수신 번호 API 레벨 제한**: Android 10+에서는 `EXTRA_INCOMING_NUMBER`를 얻으려면 `READ_CALL_LOG`도 필요하다. 권한이 없어도 발신 경로는 정상 동작하므로, 수신 매칭만 일부 기기에서 제한될 수 있다.
- **통화 실패/부재중 처리**: 통화 시간이 5초 미만이면 DB 저장 및 알림을 건너뛰어 부재중 전화나 신호음만 울린 경우를 걸러냈다.
- **pending 경합 조건**: 사용자가 pending을 설정해두고 다른 전화를 받는 경우를 방지하기 위해 pending 설정 시 타임스탬프를 함께 저장하고, OFFHOOK 시 60초 이내 pending만 사용하도록 제한했다.

### 기술적 의사결정

- **READ_CALL_LOG 포함 여부**: Play Store가 이 권한에 민감하지만, 케어 앱의 정당한 사용 사례로 포함했다. 없어도 앱 발신 경로(주 사용 시나리오)는 완전히 동작한다.
- **별도 통화 로그 테이블 vs. `call_recordings` 재활용**: `call_recordings`에 `status: "detected"`로 삽입해 기존 테이블 스키마를 재활용했다. `storage_path`가 NOT NULL 제약이면 삽입이 실패할 수 있으나, DB 저장 실패와 무관하게 알림은 항상 표시된다.
- **네이티브 HTTP 호출**: 백그라운드 BroadcastReceiver에서 Supabase JS 클라이언트를 사용할 수 없으므로 `HttpURLConnection`으로 직접 REST 요청했다.

- BroadcastReceiver 동작은 에뮬레이터의 전화 앱 시뮬레이션 또는 실기기 직접 통화로만 검증 가능
