## 세션 6: 앱 기능 전면 수정 — OAuth·데이터·UI 4개 이슈 해결

### 요청과 목표

- 빌드 완료 후 테스트에서 발견된 4가지 문제 동시 수정
  1. Google 로그인 → "invalid flow state, no valid flow state found" 오류
  2. 어르신 등록 탭에서 등록 후 홈 화면으로 돌아오면 목록이 갱신 안 됨
  3. 기존 테스트 계정으로 로그인해도 홈 화면에 어르신 목록이 안 보임
  4. 홈 화면 낮/밤 모드 전환 버튼과 기능별 바로가기 그리드 메뉴 누락

### 구현 및 변경 내용

**1. Google OAuth "invalid flow state" 수정**

기존 구조는 `login.tsx`에서 `WebBrowser.openAuthSessionAsync` 완료 후 `exchangeCodeForSession`을 직접 호출하는 방식이었다. 이 방식은 Android에서 브라우저 복귀 직후 AsyncStorage의 PKCE code verifier가 아직 준비되지 않은 시점에 교환을 시도해 state mismatch 오류를 유발했다.

해결책: `exchangeCodeForSession` 호출을 `login.tsx`에서 제거하고, 대신 `_layout.tsx`에 `Linking.addEventListener("url", ...)` 리스너를 등록해 딥링크 콜백을 받는 즉시 교환하도록 이동했다. 앱이 완전히 포그라운드로 복귀한 후 AsyncStorage 접근이 안정화된 시점에 실행되므로 state 불일치가 발생하지 않는다. cold start(앱이 닫힌 상태에서 딥링크로 열리는 경우)는 `Linking.getInitialURL()`로 별도 처리했다.

**2. 어르신 목록 미반영 수정 (`useFocusEffect`)**

`index.tsx`의 데이터 조회가 컴포넌트 마운트 시 한 번만 실행되는 `useEffect`에 묶여 있었다. 탭 간 이동 시 이미 마운트된 컴포넌트는 재실행되지 않아 등록 탭에서 돌아와도 목록이 갱신되지 않았다. `expo-router`의 `useFocusEffect`로 교체해 탭이 포커스를 얻을 때마다 Supabase에서 재조회하도록 변경했다.

**3. 다크/라이트 모드 테마 시스템 (`ThemeProvider`)**

`lib/theme-context.tsx`를 새로 만들어 LIGHT/DARK 두 가지 팔레트를 정의하고 `AsyncStorage`에 사용자 선택을 저장한다. `_layout.tsx` 최상위에 `ThemeProvider`로 앱 전체를 감싸고, 각 화면에서 `useTheme()` 훅으로 `colors`, `isDark`, `toggleTheme`에 접근한다.

홈 화면 헤더 우상단에 Sun/Moon 토글 버튼을 추가했고, 탭바(`_layout.tsx`)도 `useTheme()`를 통해 배경색·보더색·아이콘 색상이 테마에 따라 동적으로 변하도록 수정했다.

**4. 빠른 메뉴 그리드**

홈 화면 스탯 패널 아래에 8개 항목 4열 반응형 그리드를 추가했다: 통화 녹음, AI 분석, 어르신 등록, 일정 관리, 안전 알림, 통계, AI 비서, 설정. 각 항목은 아이콘 + 레이블 조합으로 해당 탭 또는 화면으로 라우팅된다.

### 문제와 해결 과정

- **"invalid flow state" 원인 분석**: OAuth PKCE 흐름에서 `expo-auth-session`이 생성한 code verifier를 AsyncStorage에 저장하는 시점과, 브라우저에서 앱으로 복귀하는 시점 사이에 타이밍 경합이 발생한다. Android Chrome Custom Tab이 닫히면서 앱이 포그라운드로 오는 직후 `exchangeCodeForSession`을 즉시 호출하면 verifier가 아직 쓰기 완료되지 않아 실패할 수 있다. Linking 이벤트 기반으로 처리를 뒤로 미루면 이 경합을 피할 수 있다.
- **`useEffect` → `useFocusEffect` 전환**: React Navigation / Expo Router에서 탭은 처음 방문 후 언마운트되지 않기 때문에 `useEffect`는 최초 마운트 한 번만 실행된다. 탭 포커스 복귀 시 재실행이 필요한 데이터 조회는 반드시 `useFocusEffect`를 사용해야 한다.

### 기술적 의사결정

- **Linking 기반 OAuth vs. WebBrowser 반환값 기반**: WebBrowser의 반환값에서 URL을 파싱해 `exchangeCodeForSession`을 호출하는 방식이 더 단순하지만 Android에서 타이밍 문제가 발생한다. Linking 이벤트 기반은 코드가 약간 분산되지만 신뢰성이 높다.
- **테마 Context vs. 외부 라이브러리**: `react-native-paper`, `tamagui` 등 UI 라이브러리 없이 최소한의 Context + AsyncStorage로 구현해 번들 크기 증가 없이 다크/라이트 전환을 제공했다.

### 테스트 및 검증

- 로컬 코드 변경만 완료된 상태이며, 실제 앱에서의 동작 검증은 versionCode 5 EAS 빌드 완료 후 가능하다.
- TypeScript 타입 오류 없이 저장되었고 컴파일 오류 없음(런타임 테스트 미완료).

### 변경 파일 (silverlink-mobile)

- `lib/theme-context.tsx`: LIGHT/DARK 팔레트, ThemeProvider, useTheme 훅 — 신규 생성
- `app/_layout.tsx`: Linking 딥링크 핸들러 추가, ThemeProvider 래핑
- `app/(auth)/login.tsx`: Google OAuth에서 `exchangeCodeForSession` 제거, browser 열기만 담당
- `app/(tabs)/index.tsx`: `useFocusEffect` 데이터 재조회, 빠른 메뉴 그리드, 다크/라이트 토글 버튼
- `app/(tabs)/_layout.tsx`: `useTheme()` 연동으로 탭바 동적 테마 적용

### 미해결 사항 및 다음 작업

- EAS versionCode 5 빌드 제출 후 실제 기기에서 4가지 수정 검증 필요

## 세션 7: 네이티브 안드로이드 백그라운드 통화 감지 구현

### 요청과 목표

- 앱이 백그라운드에 있을 때도 등록된 어르신과의 통화를 자동으로 감지하고 싶다는 사용자 요청
- 통화 종료 후 통화 메타데이터(어르신 ID, 통화 시간)를 Supabase에 자동 저장
- 통화 후 삼성 AI 요약 공유를 유도하는 푸시 알림 표시
- 앱에서 "전화 연결" 버튼으로 발신할 때도 동일하게 기록

### 구현 및 변경 내용

**아키텍처: 두 가지 통화 경로 처리**

1. **앱에서 발신 (주 경로)**: 어르신 카드 팝업의 "전화 연결" 버튼 → JS에서 `setPendingOutgoingCall(elderId, elderName)` 호출 → SharedPreferences에 60초 유효 pending 저장 → 안드로이드가 OFFHOOK 감지 시 pending을 active로 승격 → IDLE(통화 종료) 시 DB 저장 + 알림
2. **어르신 수신 전화**: BroadcastReceiver가 RINGING 시 `EXTRA_INCOMING_NUMBER`를 읽어 등록된 어르신 번호와 매칭 → OFFHOOK 시 통화 시작 시간 기록 → IDLE 시 처리

**네이티브 코틀린 파일 (신규)**

- `CallStateReceiver.kt`: `TelephonyManager.ACTION_PHONE_STATE_CHANGED` BroadcastReceiver. RINGING → OFFHOOK → IDLE 상태 전이 추적. Supabase REST API(`/rest/v1/call_recordings`)에 `status: "detected"`로 직접 HTTP POST. `NotificationCompat`으로 통화 완료 알림 표시
- `CallDetectionModule.kt`: React Native 네이티브 모듈. `syncElders(eldersJson, token, url, key, userId)`, `setPendingOutgoingCall(elderId, name)` 두 메서드를 `@ReactMethod`로 노출. SharedPreferences에 저장
- `CallDetectionPackage.kt`: `ReactPackage` 구현체로 `CallDetectionModule` 등록

**기존 파일 수정**

- `MainApplication.kt`: `add(CallDetectionPackage())` 추가
- `AndroidManifest.xml`: `READ_PHONE_STATE`, `READ_CALL_LOG`, `POST_NOTIFICATIONS` 권한 추가; `<receiver android:name=".CallStateReceiver" android:exported="true">` 등록
- `app.json`: `android.permissions` 배열에 3개 권한 추가
- `src/modules/call-detection.ts` (신규): JS 브릿지. `requestCallDetectionPermissions()`, `syncElders()`, `setPendingOutgoingCall()` 함수 export
- `app/_layout.tsx`: `requestCallDetectionPermissions()` 세션 시작 시 호출
- `app/(tabs)/index.tsx`: `fetchData` 완료 후 `syncElders()` 호출; 전화 연결 버튼에 `setPendingOutgoingCall()` 추가
- `android/app/build.gradle`: versionCode 9 → 10, versionName 1.0.8 → 1.0.9

### 문제와 해결 과정

- **발신 시 번호 불명 문제**: Android 10+에서 `ACTION_PHONE_STATE_CHANGED` 브로드캐스트에 발신 번호가 포함되지 않는다. 앱에서 전화를 걸기 직전에 어르신 정보를 SharedPreferences에 저장하는 `setPendingOutgoingCall` 방식으로 우회했다.
- **수신 번호 API 레벨 제한**: Android 10+에서는 `EXTRA_INCOMING_NUMBER`를 얻으려면 `READ_CALL_LOG`도 필요하다. 권한이 없어도 발신 경로는 정상 동작하므로, 수신 매칭만 일부 기기에서 제한될 수 있다.
- **통화 실패/부재중 처리**: 통화 시간이 5초 미만이면 DB 저장 및 알림을 건너뛰어 부재중 전화나 신호음만 울린 경우를 걸러냈다.
- **pending 경합 조건**: 사용자가 pending을 설정해두고 다른 전화를 받는 경우를 방지하기 위해 pending 설정 시 타임스탬프를 함께 저장하고, OFFHOOK 시 60초 이내 pending만 사용하도록 제한했다.

### 기술적 의사결정

- **READ_CALL_LOG 포함 여부**: Play Store가 이 권한에 민감하지만, 케어 앱의 정당한 사용 사례로 포함했다. 없어도 앱 발신 경로(주 사용 시나리오)는 완전히 동작한다.
- **별도 통화 로그 테이블 vs. `call_recordings` 재활용**: `call_recordings`에 `status: "detected"`로 삽입해 기존 테이블 스키마를 재활용했다. `storage_path`가 NOT NULL 제약이면 삽입이 실패할 수 있으나, DB 저장 실패와 무관하게 알림은 항상 표시된다.
- **네이티브 HTTP 호출**: 백그라운드 BroadcastReceiver에서 Supabase JS 클라이언트를 사용할 수 없으므로 `HttpURLConnection`으로 직접 REST 요청했다.

### 테스트 및 검증

- TypeScript/Kotlin 문법 수준에서 오류 없이 작성. 런타임 동작은 EAS 빌드(versionCode 10) 후 실기기 테스트 필요
- BroadcastReceiver 동작은 에뮬레이터의 전화 앱 시뮬레이션 또는 실기기 직접 통화로만 검증 가능

### 변경 파일 (silverlink-mobile)

- `android/app/src/main/java/ai/silverlink/mobile/CallStateReceiver.kt`: 신규 — BroadcastReceiver
- `android/app/src/main/java/ai/silverlink/mobile/CallDetectionModule.kt`: 신규 — React Native 네이티브 모듈
- `android/app/src/main/java/ai/silverlink/mobile/CallDetectionPackage.kt`: 신규 — 패키지 등록
- `android/app/src/main/java/ai/silverlink/mobile/MainApplication.kt`: CallDetectionPackage 추가
- `android/app/src/main/AndroidManifest.xml`: 권한 3개 + Receiver 등록
- `app.json`: android 권한 배열 갱신
- `src/modules/call-detection.ts`: 신규 — JS 브릿지
- `app/_layout.tsx`: 권한 요청 추가
- `app/(tabs)/index.tsx`: syncElders, setPendingOutgoingCall 연동
- `android/app/build.gradle`: versionCode 10, versionName 1.0.9

### 미해결 사항 및 다음 작업

- `call_recordings` 테이블 `storage_path` NOT NULL 제약 여부 확인 필요 — 제약 있으면 별도 `call_logs` 테이블 생성 또는 컬럼 nullable 변경 필요
- versionCode 10 EAS 빌드 후 실기기에서 통화 감지 검증

- 추후: 감지된 통화 기록을 홈 화면 통계(이번 달 통화 횟수 등)에 반영

### 커밋 정보

- 커밋 메시지: `feat: 네이티브 안드로이드 백그라운드 통화 감지 — CallStateReceiver, CallDetectionModule`
- 커밋 해시: `282901b` (silverlink-mobile)

## 세션 11: 모바일 앱 반응형 UI 적용

### 요청과 목표

- 화면 폭이 다른 안드로이드 기기에서도 레이아웃이 자연스럽게 동작하도록 반응형 처리
- 접근성 폰트 확대 설정 시 레이아웃이 깨지지 않도록 보호

### 구현 및 변경 내용

**메인 화면 메뉴 그리드 동적 너비 (`index.tsx`)**

기존 `width: "22%"` + `flexGrow: 1` 방식은 비율 기반이라 화면 폭이 크거나 작을 때 아이템 간격이 불균일해질 수 있었다. `useWindowDimensions()`로 현재 화면 폭을 읽어 열 수와 아이템 너비를 정확하게 계산하는 방식으로 교체했다.

```tsx
const { width: screenW } = useWindowDimensions();
const MENU_COLS = screenW < 360 ? 3 : 4;  // 360px 미만 소형폰은 3열
const menuItemW = Math.floor((screenW - 32 - (MENU_COLS - 1) * 10) / MENU_COLS);
```

- `screenW - 32`: 좌우 패딩(16px × 2) 제거
- `(MENU_COLS - 1) * 10`: 아이템 간 gap 합산 제거
- 나머지를 열 수로 나눠 정수로 버림

StyleSheet에서는 `width: "22%"`와 `flexGrow: 1`을 제거하고, 런타임에 계산된 `{ width: menuItemW }`를 인라인 스타일로 전달한다.

**통계 숫자 자동 축소 (`index.tsx`, `tasks.tsx`, `stats.tsx`)**

헤더 통계 패널의 숫자(등록 어르신, 이번 달 분석, 주의 신호 등)에 `adjustsFontSizeToFit` + `numberOfLines={1}`을 적용했다. 숫자가 길어지거나 화면이 좁을 때 폰트를 자동으로 줄여 넘침을 방지한다.

**소형 텍스트 접근성 폰트 보호**

메뉴 라벨(fontSize 11)에 `maxFontSizeMultiplier={1.2}`, 서브 라벨(fontSize 10)에 `maxFontSizeMultiplier={1.0}`을 적용했다. Android 접근성 설정에서 폰트 크기를 크게 올려도 레이아웃이 깨지지 않는다.

**서브 화면 헤더 타이틀 보호 (6개 화면)**

`alerts, tasks, stats, assistant, deliveries, references` 6개 화면의 헤더 타이틀 `<Text>`에 `numberOfLines={1}` + `adjustsFontSizeToFit`을 추가했다. "사회적 연결 점수"처럼 긴 타이틀이 작은 화면에서 잘리거나 줄 바꿈되는 것을 방지한다.

### 문제와 해결 과정

- **`flexGrow: 1` 충돌**: 명시적 `width: menuItemW`와 `flexGrow: 1`이 함께 있으면 마지막 줄의 2개 아이템이 비정상적으로 늘어날 수 있다. `flexGrow`를 제거하고 순수 계산 너비만 사용했다. 10개 아이템이 4열 그리드에서 마지막 줄에 2개가 남을 때 남은 공간은 빈 여백으로 두는 것이 더 자연스럽다.
- **`minHeight` 유지**: `minHeight: 90`은 그대로 두어 아이콘 + 라벨 + 서브 텍스트가 3단으로 쌓여도 최소 높이를 보장했다.

### 기술적 의사결정

- **`useWindowDimensions` vs. `Dimensions.get("window")`**: 기기 회전이나 멀티윈도우 시 동적 업데이트가 되는 `useWindowDimensions` 훅이 정적인 `Dimensions.get`보다 적합하다. 현재는 세로 모드 고정이지만 향후 확장성을 위해 훅 방식으로 선택했다.
- **360px 기준 3열 전환**: 국내 보급형 안드로이드(Galaxy A시리즈 등)의 최소 폭이 360~375px 수준이다. 360px 미만인 320px 대 초소형 기기에서 4열이면 아이템 폭이 64px 수준으로 너무 좁아 3열로 전환했다.

### 구현 및 변경 내용

**음성 재생 수정 (`record.tsx`)**

`expo-audio`의 `createAudioPlayer`는 인스턴스를 생성하자마자 `play()`를 호출하면 Android에서 음소거 상태(silent/media volume)로 실행되거나 무시되는 경우가 있다. `setAudioModeAsync({ allowsRecording: false, playsInSilentMode: true })`를 `createAudioPlayer` 호출 이전에 await해야 하며, `player.play()` 역시 await해야 오류를 캐치할 수 있다. 기존 코드는 두 호출을 await하지 않았다.

재생 완료 감지 로직도 개선했다. 기존에는 `status.isLoaded`로 판단했으나 expo-audio 1.x에서 해당 필드가 없다. `!status.playing && status.currentTime > 0 && status.currentTime >= status.duration`으로 재생 종료를 감지하도록 변경했다.

**자동 감지 배지 UI (`record.tsx`)**

`storage_path`가 없는 "detected" 레코드에 기존에는 비활성화된 재생 버튼이 표시됐다. 음성 파일이 없는 레코드에 재생 버튼이 있으면 사용자가 버그로 오인한다. "📵 자동 감지" 텍스트 배지로 교체해 음성이 없는 통화 기록임을 명확히 했다.

**기본 전화앱 발신 통화 감지 (`CallStateReceiver.kt`)**

기존 구조는 앱의 "전화 연결" 버튼을 통해 발신할 때만 `setPendingOutgoingCall`이 SharedPreferences에 어르신 정보를 저장했다. 사용자가 기본 전화앱에서 직접 전화를 걸면 pending이 설정되지 않아 통화가 감지되지 않는 문제였다.

해결책: `EXTRA_STATE_IDLE` 처리 시 `elderId`가 없으면(앱 발신 pending도, 수신 매칭도 없었던 경우) `findElderFromCallLog()`를 호출한다. 이 함수는 `CallLog.Calls.CONTENT_URI`를 최신순으로 조회해 가장 최근 통화 번호를 꺼내고, 등록된 어르신의 번호와 뒤 9자리를 비교해 매칭한다. 발신(OUTGOING_TYPE) 또는 수신 완료(INCOMING_TYPE)만 처리하고 부재중(MISSED_TYPE)은 건너뛴다.

### 문제와 해결 과정

- **발신 번호 접근 제한**: Android 10+에서 `ACTION_PHONE_STATE_CHANGED` 브로드캐스트에 발신 번호가 포함되지 않는다. 통화 종료(IDLE) 시점에 이미 콜로그에 기록된 번호를 읽는 방식으로 우회했다. `READ_CALL_LOG` 권한은 이전 세션에서 이미 매니페스트에 등록돼 있었다.
- **타이밍**: IDLE 이벤트 발생 시 삼성 콜로그에 통화 기록이 이미 기록돼 있어 추가 지연 없이 조회 가능했다.

### 기술적 의사결정

- **콜로그 조회 시점**: OFFHOOK 대신 IDLE(통화 종료) 시점에 조회한다. OFFHOOK 시에는 아직 어떤 통화가 됐는지 확정되지 않고, IDLE 시에는 콜로그에 완성된 레코드가 있기 때문이다.
- **부재중 제외**: MISSED_TYPE은 실제 통화가 이루어지지 않았으므로 처리 대상에서 제외. 이미 durationSec < 5 조건으로 걸러지지만 타입 레벨에서도 명확히 제외했다.

### 변경 파일 (silverlink-mobile)

- `android/app/src/main/java/ai/silverlink/mobile/CallStateReceiver.kt`: `findElderFromCallLog()` 함수 추가, IDLE 핸들러에서 elderId 없을 때 호출, `showNotification` 시그니처에 elderId 추가
- `app/(tabs)/record.tsx`: `setAudioModeAsync` await 순서 수정, `player.play()` await 추가, 재생 종료 판단 로직 수정, 자동 감지 배지 UI 추가
- `android/app/build.gradle`: versionCode 22, versionName "1.0.21"

### 테스트 및 검증

- `bundleRelease` (AAB versionCode 22): `BUILD SUCCESSFUL in 26s`

## 세션 13: 통화 녹음 파일 가져오기 + share-import 화면 개선 (versionCode 23)

### 요청과 목표

- "보내기" 알림 버튼을 눌렀을 때 '통화 요약 가져오기' 화면만 뜨고 실제 녹음 파일이 저장되지 않는 문제 해결
- 녹음 기록 화면에서 "detected" 상태가 영문 그대로 표시되는 문제 → 한글로 변경
- 녹음 탭으로 돌아왔을 때 목록이 자동 갱신되지 않는 문제 해결

### 구현 및 변경 내용

**share-import 화면 전면 개선 (`share-import.tsx`)**

기존 화면은 삼성 AI 통화 요약 텍스트를 붙여넣어 분석하는 단일 기능이었다. "보내기" 버튼을 눌러 이 화면으로 왔을 때 사용자는 녹음 파일이 자동으로 저장되기를 기대했으나 실제로는 텍스트 입력 폼만 있었다.

`expo-document-picker`를 설치하고 화면 상단에 "🎙️ 녹음 파일 가져오기" 섹션을 추가했다. 사용자가 "📂 녹음 파일 선택" 버튼을 누르면 `getDocumentAsync({ type: "audio/*" })`로 파일 선택창이 열린다. 선택된 파일은 `expo-file-system`의 `File.arrayBuffer()`로 읽어 Supabase `call-recordings` 스토리지 버킷에 업로드하고 `call_recordings` 테이블에 `status: "pending"`으로 삽입한다. 저장 성공 시 "녹음 목록 보기" 버튼이 있는 알림창을 표시해 `/(tabs)/record`로 바로 이동할 수 있다.

알림 딥링크 파라미터 연동: `CallStateReceiver.kt`의 "보내기" 딥링크를 `silverlink:///(tabs)/share-import?parentId={UUID}`로 변경해 통화한 어르신이 화면 진입 시 자동 선택되도록 했다. `useLocalSearchParams()`로 `parentId`를 읽고, `parents` 목록이 로드된 후 일치하는 어르신을 자동 선택한다.

화면 포커스 시 어르신 목록 새로고침: `useFocusEffect`를 추가해 다른 탭에서 돌아왔을 때도 최신 목록이 표시된다.

**녹음 기록 화면 개선 (`record.tsx`)**

- `statusLabel()` 함수에 `"detected"` 케이스를 추가해 "통화 감지" 한글로 표시 (보라색 배지)
- `useFocusEffect` 추가: 탭에 포커스가 올 때마다 `loadRecordings()` 호출. "녹음 목록 보기" 버튼으로 돌아왔을 때 방금 업로드한 파일이 바로 목록에 보임

### 문제와 해결 과정

- **자동 저장 방식 vs. 파일 선택 방식**: 삼성 전화앱이 저장한 녹음 파일을 앱에서 자동으로 찾으려면 `READ_MEDIA_AUDIO` 권한과 삼성 별 폴더 경로(`/storage/emulated/0/Recordings/Call/` 등)가 필요하다. 경로가 One UI 버전마다 다르고 타이밍 문제도 있어 안정성이 낮다. 파일 선택 방식은 1~2탭이 더 필요하지만 안정적이고 다른 앱의 녹음도 가져올 수 있어 채택했다.
- **`expo-document-picker` 미설치**: `npx expo install expo-document-picker`로 설치. `package.json`에 `"expo-document-picker": "~14.0.8"` 추가됨.

### 기술적 의사결정

- **업로드 후 DB 삽입 (pending 상태)**: `storage_path` 없이 DB 레코드만 만드는 것은 의미가 없고, 반대로 DB 없이 파일만 올리면 목록에 뜨지 않는다. 파일 업로드 성공 후에만 DB 삽입하는 순차 처리로 원자성을 최대한 확보했다.
- **`useFocusEffect` 위치**: `share-import`와 `record` 양쪽에 모두 추가해, 어느 경로로 진입하더라도 항상 최신 데이터를 보여주도록 했다.
