# 2026-07-22 개발일지

## 오늘의 요약

오늘은 SilverLink AI 모바일 앱과 웹 대시보드 전반의 UI/UX를 "바이브코딩 MVP 느낌"에서 "실제 제품처럼 보이는 비의료 돌봄 서비스"로 전환하는 대규모 리디자인 작업을 수행했다. 모바일은 디자인 토큰 도입, 5개 탭 재구성, 8개 화면 정리, 5개 공통 컴포넌트 생성까지 완료했고, 웹은 12개 기능 그리드를 상태 중심 대시보드로 재구성하고 용어 일관성을 정리했다. 또한 통화 종료 후 "요약할까요?" 푸시 알림을 제거하고 자동 AI 분석 흐름으로 전환하는 기능 변경도 포함되었으며, 모바일 versionCode를 46으로 올려 UI 리디자인이 반영된 AAB 빌드까지 완료했다.

## 세션 1: 모바일 통화 종료 후 자동 분석 흐름 전환 (versionCode 45)

### 요청과 목표
- 통화 종료 후 "요약할까요?" 푸시 알림 대신 자동으로 AI 분석 API를 호출하도록 변경
- 사용자가 별도 탭을 하지 않아도 분석이 백그라운드에서 진행되는 경험

### 구현 및 변경 내용
- `CallStateReceiver.kt`: `showNotification()` 제거 → 업로드 성공 시 `autoAnalyzeRecording()` 호출
  - POST `/api/recordings/analyze` (Bearer 토큰, 3분 타임아웃)
  - 알림 관련 import 전부 제거 (NotificationChannel, NotificationManager, PendingIntent, NotificationCompat, Build)
- `CallDetectionModule.kt`: `syncElders()` 에 `apiBaseUrl` 파라미터 추가
- `src/modules/call-detection.ts`: JS 래퍼 함수 시그니처 업데이트
- `app/(tabs)/index.tsx`: `syncElders()` 호출에 `process.env.EXPO_PUBLIC_API_URL` 전달
- `android/app/build.gradle`: versionCode 44 → 45, versionName "1.0.43" → "1.0.44"

### 기술적 의사결정
- 분석 실패 시 사용자에게 별도 알림 없음 — 실패해도 DB에 `status: "failed"` 기록되어 앱 내 기록 화면에서 확인 가능
- 녹음 파일이 감지되지 않으면 자동 분석 미실행 (share-import 수동 업로드 경로 유지)

### 변경 파일

## 세션 9: 중대 오류(Crash) 수정 — versionCode 50

### 오류 증상
- versionCode 49 설치 후 앱이 잠깐 띄워졌다가 즉시 꺼짐

### 원인 분석
- `ElderDetailModal`에 추가한 `useRef([0,1,2,3].map(() => new Animated.Value(0)))` 패턴이 문제
- 해당 컴포넌트는 `elder=null` 일 때 `return null` (아무것도 렌더링 안 함)
- **그러나 훅은 early return 이전에 실행** → `Animated.Value` 4개가 생성되어 native driver에 등록됨
- `newArchEnabled=true` (Fabric/JSI/New Architecture) 환경에서 native driver가 연결할 View를 찾지 못해 crash
- Metro 번들/캐시 문제가 아닌 **코드 로직 문제**로 확인

### 수정 내용 (versionCode 50)
- `index.tsx`: `Animated`, `useRef`, `useEffect` 임포트 제거
- `ElderDetailModal`: `Animated.Value` 직접 사용 방식 제거
- **대체 방식**: 이미 안정 동작이 검증된 `StaggerView` 컴포넌트 재사용
  - 모달 4개 섹션(헤더/정보/액션/삭제버튼)을 각각 `<StaggerView index={0~3}>` 로 감쌈
  - `key={`header-${elder.id}`}` prop → 다른 어르신 선택 시 StaggerView 재마운트 → 애니메이션 재실행
  - `if (!elder) return null` 이후에 렌더링되므로 null 렌더 시 훅 실행 없음 (안전)

### 검증
- `npx tsc --noEmit` → EXIT:0
- `./gradlew bundleRelease` → BUILD SUCCESSFUL (3분 47초, Metro 캐시 초기화 후 전체 재빌드)

### 변경 파일
- `app/(tabs)/index.tsx`
- `android/app/build.gradle` (versionCode 50)

### 구현 및 변경 내용

**모바일 크래시 근본 원인 (2가지):**

1. `expo-splash-screen@57.0.4` 버전 불일치
   - Expo SDK 54가 요구하는 버전은 `~31.0.13`인데 57.0.4가 설치되어 있어 네이티브 바인딩 오류 → 크래시
   - `package.json`: `"expo-splash-screen": "^57.0.4"` → `"^31.0.13"` 으로 다운그레이드

2. `expo-updates` OTA 캐시 충돌
   - `EXPO_UPDATES_CHECK_ON_LAUNCH=ALWAYS` 설정으로 이전 네이티브 빌드용 OTA 번들을 로드 → 새 네이티브와 불일치 → 크래시
   - `AndroidManifest.xml`: `ENABLED=false`, `CHECK_ON_LAUNCH=NEVER` 로 업데이트 완전 비활성화
   - `app.json`: `"enabled": false` 추가, `runtimeVersion` 제거

**남색 화면 완전 제거 (로딩 화면 대체):**
- 기존: 네이티브 스플래시 숨긴 뒤 JS가 준비되는 동안 남색 빈 화면 1초 노출

- 네이티브 스플래시(파란색) → React 오버레이(동일 파란색) → 로그인/홈 화면으로 끊김없이 전환
