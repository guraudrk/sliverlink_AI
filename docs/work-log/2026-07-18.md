## 세션: 블랙아웃 근본 수정 + 챗봇 답변 버그 픽스 (versionCode 35→39)

### 요청과 목표

- AI 챗봇 대화 중 화면이 계속 블랙아웃되는 문제 해결
- 삼성 통화 녹음 자동 인식 개선 및 수동 파일 첨부 기능 추가
- 결과 보기 화면 구조 개선 (웹 스크린샷 기준)
- AAB를 기본 빌드 방식으로 변경

### 구현 및 변경 내용

**1. 블랙아웃 문제 — 3단계 접근 후 최종 해결**

처음엔 삼성 기기의 화면 자동 꺼짐 타이머가 원인이라고 판단하고 `expo-keep-awake` 패키지를 도입했다. 그러나 두 번의 구현 방식 변경에도 증상이 지속됐고, 최종적으로 `MainActivity.kt`에서 Android 네이티브 레벨로 직접 `FLAG_KEEP_SCREEN_ON` 플래그를 설정해 해결됐다.

- 1차 시도: `activateKeepAwakeAsync()` / `deactivateKeepAwake()` 함수 방식 → 내부에서 `ExpoKeepAwake.activate?.()` optional chaining으로 네이티브 모듈 미로드 시 **무음 실패**
- 2차 시도: `{sending && <KeepAwakeGuard />}` 컴포넌트 방식 → 같은 네이티브 모듈 경유라 삼성 기기에서 여전히 미작동
- 최종 해결: `MainActivity.kt`의 `onCreate`에서 `window.addFlags(WindowManager.LayoutParams.FLAG_KEEP_SCREEN_ON)` 직접 설정 → JS 브릿지 없이 Android OS가 직접 처리하므로 삼성 펌웨어 영향 없음

**2. 챗봇 답변 `[object Object]` 버그 수정**

RAG API (`/api/rag/ask`) 엔드투엔드 테스트를 작성해 실행한 결과, `json.answer`가 `{ answerText, evidence, nextSteps }` 객체임을 확인. 모바일에서 `json.answer`를 그대로 `<Text>`에 넣어 `[object Object]`가 표시되고 있었음.

- 수정: `json.answer` → `json.answer?.answerText ?? json.answer`

**3. RAG API 엔드투엔드 테스트 스크립트 작성**

`test-rag.mjs` 작성 — Supabase 로그인 → Bearer 토큰 획득 → POST `/api/rag/ask` → 응답 확인. 서버 정상(405), 인증 차단(401), 로그인 성공, AI 답변 3초 내 수신 모두 확인.

**4. package.json 빌드 스크립트 추가**

```json
"build:aab": "cd android && gradlew bundleRelease",
"build:apk": "cd android && gradlew assembleRelease"
```

**5. Play Store 버전 충돌로 인한 versionCode 연속 증가**

APK와 AAB를 혼재 업로드하면서 Play Store에서 "버전 코드 이미 사용됨" 오류 반복 발생 → 35→36→37→38→39 순으로 증가.

### 문제와 해결 과정

- **expo-keep-awake 무음 실패**: `activate?.()` optional chaining이 네이티브 모듈 미초기화 시 에러 없이 no-op 처리 → 문제를 발견하기 어려움. JS 레벨 패키지보다 네이티브 레벨 직접 처리가 더 신뢰성 높음을 확인

### 구현 시도 및 변경 과정

`react-native`의 `Linking.openURL(API_URL + step.href)`로 외부 브라우저에서 웹 대시보드(`https://silverlink-ai.vercel.app/dashboard/parents/{id}`)를 열도록 구현. 기술적으로는 동작하나, 사용자가 원하는 것은 앱 내 화면 이동임을 확인해 방향을 수정.

웹 `/dashboard/parents/{parentId}` 페이지 구조를 분석한 결과(사회 연결 점수·미확인 안전 알림·AI 케어 플랜·일정·통화 기록), 모바일에서 가장 관련성 높은 화면은 `안전 알림` 탭(`alerts.tsx`)임을 판단.

```tsx
router.push({ pathname: "/(tabs)/alerts", params: { elderId } });
```

`alerts.tsx`에 `useLocalSearchParams`로 `elderId`를 받아 Supabase 쿼리에 `.eq("elder_id", elderId)` 필터 추가. 그러나 이 프로젝트의 Expo Router 설정에서 `router.push` 객체 형태가 타입 오류로 **조용히 실패**하는 문제가 있었음.

**v44: router.push 문자열 형태 — 최종 해결**

`index.tsx`가 모든 숨겨진 탭 이동에 `router.push(href as any)` 문자열 형태를 쓰는 패턴을 발견하고 동일하게 적용:

```tsx
router.push(`/(tabs)/alerts?elderId=${elderId}` as any);
```

`useLocalSearchParams<{ elderId?: string }>()`로 쿼리스트링 파라미터를 읽고 Supabase 필터 적용. 헤더에 "AI 비서 연결 — 해당 어르신 알림" 배지 추가.

### 기술적 의사결정

- **이동 대상 화면**: 웹 어르신 상세 페이지에 해당하는 네이티브 화면이 없어, 가장 관련 콘텐츠인 안전 알림 탭으로 이동하는 것이 최선으로 판단. 향후 전용 어르신 상세 화면 개발 시 수정 가능.
- **elderId 전달 방식**: Expo Router 탭 간 파라미터 전달은 URL 쿼리스트링 형태(`?elderId=xxx`)가 가장 신뢰성 높음. 탭 스크린에서 `useLocalSearchParams`가 navigation 시 값을 반응형으로 업데이트함.
- **elder_id 필터링**: `safety_alerts` 테이블은 `elder_id` 컬럼으로 어르신을 구분. 웹과 동일한 Supabase 쿼리 패턴 적용.
