# Claude Code 커밋 연동 개발일지 지침

이 문서는 Claude Code가 커밋·푸시 작업을 수행할 때마다 개발일지를 자동으로 갱신하도록 만드는 영구 지침이다.

## 1. 최초 설정

이 문서를 처음 전달받은 Claude Code는 다음 작업을 즉시 수행한다.

1. 이 파일을 프로젝트의 `docs/CLAUDE_COMMIT_WORKLOG_INSTRUCTIONS.md`에 저장한다.
2. 프로젝트 루트의 `CLAUDE.md`가 없으면 생성한다.
3. `CLAUDE.md`에 아래 영구 규칙을 추가한다. 동일한 규칙이 이미 있다면 중복 추가하지 않는다.

```md
## 커밋·푸시 전 개발일지 자동 갱신

- 사용자가 커밋 또는 푸시를 요청하면, 실제 `git commit`을 실행하기 전에 반드시 `docs/CLAUDE_COMMIT_WORKLOG_INSTRUCTIONS.md`를 읽고 따른다.
- 해당 지침에 따라 오늘 날짜의 `docs/work-log/YYYY-MM-DD.md`를 생성하거나 갱신한다.
- 같은 날짜의 파일이 이미 있으면 새 파일을 만들지 말고, 기존 내용과 중복되지 않게 이번 작업을 추가한다.
- 개발일지와 `README.md`의 작업일지 색인을 코드 변경과 같은 커밋에 포함한다.
- 개발일지 갱신이 실패하면 커밋·푸시를 진행하지 말고 실패 원인을 사용자에게 알린다.
```

4. `docs/work-log/` 폴더가 없으면 생성한다.
5. `README.md`에 `## 개발일지` 섹션이 없으면 만들고 `docs/work-log/`의 날짜별 문서 링크를 최신 날짜순으로 표시한다.

## 2. 실행 시점

다음 요청이 들어오면 사용자가 별도로 개발일지 갱신을 말하지 않아도 이 지침을 자동 실행한다.

- 커밋해줘, 커밋하자, commit
- 푸시해줘, push
- 변경사항 저장하고 올려줘
- PR을 만들어줘
- 작업을 마무리해줘
- 그 밖에 현재 변경을 Git 이력에 남기는 요청

일반적인 코드 작성·분석·질문 단계에서는 매 응답마다 일지를 수정하지 않는다. **커밋 직전 한 번** 갱신하는 것을 원칙으로 한다.

## 3. 분석해야 할 근거

개발일지는 추측이 아니라 다음 근거를 종합해 작성한다.

1. 이번 Claude Code 세션에서 사용자가 요청한 내용
2. Claude가 제안하거나 실행한 접근 방법
3. 실제 `git status`, `git diff`, `git diff --stat`
4. 변경한 소스코드·설정·문서
5. 발생한 오류와 실패한 시도
6. 사용자가 선택하거나 거부한 대안
7. 실행한 테스트·빌드·린트와 그 결과
8. 아직 해결하지 못한 문제와 다음 작업

대화에서 언급됐지만 코드나 실행 결과로 확인되지 않은 성과는 완료했다고 단정하지 않는다.

## 4. 날짜와 파일 규칙

- 날짜 기준: `Asia/Seoul`
- 파일 경로: `docs/work-log/YYYY-MM-DD.md`
- 제목: `# YYYY-MM-DD 개발일지`
- 같은 날 여러 번 커밋하면 동일 파일에 세션 단위 섹션을 추가한다.
- 세션 제목은 작업의 핵심 결과를 20자 내외로 요약한다.
- 동일 작업을 amend하거나 재커밋할 때는 중복 문단을 추가하지 않고 기존 내용을 갱신한다.

## 5. 날짜별 일지 형식

```md
# YYYY-MM-DD 개발일지

## 오늘의 요약

오늘 진행한 작업을 비전공자도 이해할 수 있는 자연스러운 한국어 줄글 1~3문단으로 정리한다. 무엇을 왜 바꿨고 어떤 결과를 만들었는지가 드러나야 한다.

## 세션: 핵심 작업명

### 요청과 목표

- 사용자가 요청한 핵심 작업
- 작업을 시작한 이유와 기대 결과

### 구현 및 변경 내용

- 실제 구현한 기능
- 변경한 구조·로직·설정
- 주요 변경 파일과 각 파일의 역할

### 문제와 해결 과정

- 발생한 오류 또는 막힌 지점
- 원인에 대한 가설
- 실패하거나 보류한 접근과 그 이유
- 최종 해결 방법과 선택 이유

### 기술적 의사결정

- 검토한 대안
- 선택한 방식과 트레이드오프
- 유지보수·성능·보안에 미치는 영향

### 테스트 및 검증

- 실행한 명령 또는 테스트
- 성공·실패 결과
- 실행하지 못한 검증과 그 이유

### 변경 파일

- `path/to/file`: 변경 목적

### 미해결 사항 및 다음 작업

- 남아 있는 문제
- 다음 세션의 최우선 작업

### 커밋 정보

- 커밋 메시지: 커밋 후 실제 메시지 기록
- 브랜치: 현재 브랜치
```

## 6. README 색인 규칙

`README.md`의 `## 개발일지` 섹션에는 본문 전체를 복사하지 않고 링크만 유지한다.

```md
## 개발일지

- [2026-07-15 개발일지](docs/work-log/2026-07-15.md)
- [2026-07-14 개발일지](docs/work-log/2026-07-14.md)
```

- 최신 날짜를 위에 둔다.
- 같은 날짜 링크를 중복 생성하지 않는다.
- 기존 README의 다른 섹션은 삭제하거나 재작성하지 않는다.

## 7. 보안 및 품질 규칙

다음 내용은 개발일지에 절대 기록하지 않는다.

- API 키, 액세스 토큰, 비밀번호, 쿠키, 인증 헤더
- `.env` 및 비밀 설정값
- 개인정보, 이메일, 전화번호, 주소
- 고객·환자·사용자의 실제 데이터
- 비공개 URL이나 내부 시스템 접속 정보
- 대화 원문 전체

민감한 값이 로그나 diff에 보이면 `[REDACTED]`로 치환한다. 파일명이나 기술적 맥락만으로 충분할 경우 실제 값을 옮기지 않는다.

다음 품질 원칙을 따른다.

- 실제로 확인된 내용만 작성한다.
- 커밋 수나 코드 줄 수를 성과로 과장하지 않는다.
- 단순한 파일 생성보다 해결한 문제와 기술적 판단을 강조한다.
- 실패와 미완료 사항도 숨기지 않는다.
- 비전공자도 이해할 수 있지만 기술 면접 근거로도 사용할 수 있게 작성한다.
- “완벽하게”, “획기적으로” 같은 근거 없는 표현을 피한다.

## 8. 커밋 실행 순서

Claude Code는 커밋·푸시 요청을 받으면 다음 순서를 지킨다.

1. 현재 브랜치와 변경 상태 확인
2. `git diff` 및 테스트 결과 분석
3. 오늘 날짜의 개발일지 생성 또는 갱신
4. README 개발일지 링크 갱신
5. 보안정보 포함 여부 점검
6. 사용 가능한 테스트·빌드·린트 실행
7. 코드와 개발일지를 함께 스테이징
8. 명확한 커밋 메시지로 커밋
9. 일지의 커밋 정보가 실제 메시지와 일치하는지 확인
10. 사용자가 푸시까지 요청했을 때만 푸시

커밋 자체가 실패하면 성공했다고 기록하지 않는다. 커밋 후 일지의 커밋 메시지를 보정하기 위해 불필요한 추가 커밋을 만들지 말고, 커밋 전에 사용할 메시지를 확정해 일지에 기록한다.

## 9. 완료 보고

커밋 또는 푸시 후 사용자에게 다음을 짧게 보고한다.

- 갱신한 개발일지 경로
- 기록한 핵심 작업
- 커밋 메시지와 커밋 해시
- 테스트 결과
- 푸시 여부

이 지침은 사용자가 명시적으로 해제하거나 변경하기 전까지 계속 적용한다.
